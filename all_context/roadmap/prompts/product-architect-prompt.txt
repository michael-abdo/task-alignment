COLUMN DEFINITIONS
==================

FEATURE-LEVEL COLUMNS (apply to parent row):

| Column | Required | Format/Constraint |
|--------|----------|-------------------|
| Feature Name (Parent) | Yes | Outcome-based label. Never a pipeline or abstraction. Use invariant concepts. |
| Feature ID (Parent) | Yes | `<DOMAIN>-F<##>` (e.g., CORE-F01, PLAT-F05). Never reuse retired IDs. |
| Feature Domain (Parent) | Yes | One of: CORE, PLAT, PSYC, CTXT, VALD, POST |
| Feature Owner(s) | No | Person(s) accountable for Feature. Populated later. Leave empty. |

Domain definitions:
- CORE - Core analytics and detection logic
- PLAT - Platform infrastructure, UI, and integrations
- PSYC - Psychometric instruments and assessments
- CTXT - Contextual adaptation and thresholds
- VALD - Validation, QA, and production readiness
- POST - Post-meeting synthesis and reporting

COMPONENT-LEVEL COLUMNS:

| Column | Required | Format/Constraint |
|--------|----------|-------------------|
| Component Name | Yes | Atomic unit, one responsibility. Replacement must not break siblings. |
| Signals | No | Leave empty. Signal definitions go in Notes column (temporary). |
| Component Type | Yes | One of: BACK, FRONT, INFRA, INTGR |
| Component # | Yes | `C<##>` (e.g., C01, C02). Represents execution order within Feature. **Each Feature must start at C01.** |
| Component ID | Yes | `<Feature ID> | C<##>` (e.g., PLAT-F01 | C01). Globally unique, never reused. |

Component Type definitions:
- BACK - Business logic and data processing
- FRONT - Frontend UI, visualization, user interaction
- INFRA - Deployment, hosting, external service connectors
- INTGR - Orchestration across multiple BACK/FRONT/INFRA components

DEPENDENCY COLUMNS:

| Column | Required | Format/Constraint |
|--------|----------|-------------------|
| BLOCKER | Yes* | Component ID(s) that must be "Done" before ANY work can begin. Empty if none. |
| Dependencies | Yes* | Component ID(s) whose outputs this Component consumes. Allows parallel design/scaffolding. Empty if none. |

Dependency ID formats (both valid):
1. Component ID: `<Feature ID> | C<##>` (e.g., CORE-F01 | C01)
2. Legacy format: `<PREFIX>-S<##>-C<##>` (e.g., INFRA-S04-C02)

PROJECT MANAGEMENT COLUMNS:

| Column | Required | Format/Constraint |
|--------|----------|-------------------|
| Parent Item | Yes | Auto-format: `<Feature ID> / <Feature Name>` |
| Success Criteria | No | Leave empty (temporary). |
| Risk Level | No | Leave empty (temporary). |

COLUMNS TO LEAVE EMPTY (populated later by project management):

| Column | Format/Constraint |
|--------|-------------------|
| Owner | Single person. Populated later. Leave empty. |
| Hours Est | Estimated effort in hours. Populated later. Leave empty. |
| Hours Actual | Actual hours spent. Populated later. Leave empty. |
| Status | One of: Not Started, In Progress, Done. Populated later. Leave empty. |
| Start Date | YYYY-MM-DD. Populated later. Leave empty. |
| End Date | YYYY-MM-DD. Populated later. Leave empty. |
| Completion Date | YYYY-MM-DD. Populated later. Leave empty. |
| Notes | Context for WHY/WHAT decisions. TEMPORARY: Also stores Signal definitions as `<EventName>: <field1> + ...` until Signals column migration complete. |

---

RULES
=====

1. Group desires into the fewest stable, outcome-based Features.
2. Each Feature must deliver clear user value.
3. Decompose each Feature into Components that:
   - Own exactly one responsibility
   - Define clear Signals (output contract)
   - Avoid hidden coupling
   - Support replacement without breaking the Feature
4. Avoid modality, technology, or implementation-first naming.
5. Prefer invariant concepts over transient tools.
6. Limit Features to 3-6 Components unless complexity demands otherwise.
7. Each Feature's Components must start at C01 and increment sequentially (C01, C02, C03...).
8. C01 must complete before C02 begins (execution order).
9. Use BLOCKER for hard dependencies (blocks all work). Use Dependencies for soft dependencies (allows parallel design).

PLACEHOLDER RULE:
If Components cannot be determined yet, use a single placeholder row:
- Component Name: "COMPONENTS TO BE PROVIDED"
- Component #: (empty)
- Component ID: `<Feature ID> |` (no C## suffix)
- Signals, Component Type: (empty)
- Success Criteria: "Decomposed into concrete Components with assigned Owners."
- BLOCKER: (empty - required)
